## 1. Swipe mode — proximity-aware usage reranking

**Files:** `src/eyetrax/app/swipe_dtw.py`, `src/eyetrax/app/keyboard_demo.py`

Previously, swipe DTW results were ranked purely by template distance and user selection history had no effect on which word was auto-accepted.

### `swipe_dtw.py`
`match()` now returns `list[tuple[str, float]]` — `(word, dtw_distance)` pairs — instead of just words. The scores are needed for proximity comparison downstream.

### `keyboard_demo.py`
After getting scored matches, results are split into two groups using `SWIPE_RERANK_TOLERANCE = 0.15` (15%):

- **Close group** — words whose DTW distance is within 15% of the best score → reranked by `usage.rerank_words()` (personal selection frequency, descending).
- **Far group** — words outside the tolerance band → kept in original DTW order, appended after the close group.

**Example:** DTW returns `[("hello", 1.0), ("help", 1.08), ("hill", 1.4)]`. Threshold = `1.0 × 1.15 = 1.15`. "hello" and "help" are close, "hill" is far. If "help" has been selected before (count ≥ 1) and "hello" has count 0, the final order becomes `["help", "hello", "hill"]` and "help" is auto-accepted.

If one match is clearly better than the rest (distance gap exceeds 15%), DTW order is preserved and usage does not override it.

## 2. Spacebar auto-selects top bigram in dwell mode

**Files:** `src/eyetrax/app/keyboard_demo.py`

The spacebar key in dwell mode now has three tiers:

| State | Action |
|---|---|
| Keys typed (word in progress) | Accept top **trie** prediction |
| No keys typed, bigram suggestions available | Accept top **bigram** suggestion |
| Neither | Insert a literal space |

Previously the second tier was missing — space between words only inserted a space character, requiring the user to open the bigram panel manually to select a next-word suggestion.

## 3. `vertical-only` calibration added to `accuracy_demo.py`

**Files:** `src/eyetrax/app/accuracy_demo.py`

The `vertical-only` calibration mode (9 centre-column points, 1D Y-only prediction) was not available in the accuracy demo. Added:

- Import of `run_vertical_only_calibration`
- `"vertical-only"` added to `--calibration` CLI choices
- Dispatch case in `_run_calibration()` routing to `run_vertical_only_calibration` with `multi_pose` and `multi_pose_d` arguments forwarded

Run with:
```
python src/eyetrax/app/accuracy_demo.py --calibration vertical-only --camera 0 --landmark-alpha 0.7 --multi-pose --multi-position
```

## 4. Fix — accuracy test crash with `vertical-only` model

**Files:** `src/eyetrax/app/accuracy_demo.py`

The 7-point accuracy test crashed when using a `vertical-only` model because the model returns a scalar Y value, while the test expected `(x, y)` pairs (shape `(7, 2)`), causing a NumPy broadcast error.

**Fix:** In the capture loop, if `gaze_estimator.vertical_only` is set, the scalar prediction is wrapped into `[gaze_estimator.vertical_center_x, float(raw)]` before being appended, giving it the correct 2D shape for error calculation and result display.
